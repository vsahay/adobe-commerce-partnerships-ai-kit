# Conceptual Guide

**Read this before anything else.** This guide explains why the toolkit works the way it does. Customers who understand the model make better decisions at every step. Customers who skip this guide tend to produce low-quality inputs and then blame the output.

---

## The Core Principle

The AI in this toolkit does not write code from a vague description. It writes code from a **precise engineering spec** (called an LLD) that was itself written against a **precise description of your codebase** (called service cards).

The quality of the generated code is directly proportional to the quality of those two inputs.

Garbage in, garbage out — but the inverse is also true: well-structured inputs produce code that slots into your project with minimal rework.

---

## How the Toolkit Thinks: A Knowledge Funnel

Before diving into steps, it helps to understand what problem the toolkit is actually solving.

To generate code that fits your project, the AI needs to know two things simultaneously:

1. **What to build** — the feature's API operations, user journey, and business rules. Adobe knows this; you do not author it.
2. **How to build it in your project** — your naming conventions, your auth patterns, your component structure, your error handling. You know this; Adobe does not.

Neither source of knowledge is sufficient on its own. The toolkit's job is to combine them.

The diagram below shows how that combination happens across four knowledge layers. This is not a sequence you walk through from top to bottom — it is a funnel where two streams of knowledge merge into an increasingly concrete output.

```
  ADOBE'S KNOWLEDGE                   YOUR KNOWLEDGE
  (what to build)                     (how to build it)

  ┌──────────────────────┐
  │  Experience Card     │  endpoints, user journey, error semantics
  │  + API Spec          │  → distilled into the Experience Card
  └──────────┬───────────┘  and API Spec that Adobe hands you
             │
             │                    ┌───────────────────────┐
             │                    │   Service Cards       │  Your codebase —
             │                    │   (you generate       │  naming, auth,
             │                    │    from your repo)    │  patterns, structure
             │                    └──────────┬────────────┘
             │                               │
             └──────────────┬────────────────┘
                            │  AI combines both streams
                            ▼
              ┌─────────────────────────────┐
              │   LLDs (Low-Level Designs)  │  Feature plan tailored
              │   (AI-generated, you        │  to YOUR codebase —
              │    review and approve)      │  files, functions, flow
              └─────────────┬───────────────┘
                            │  AI executes the plan
                            ▼
              ┌─────────────────────────────┐
              │   Generated Code            │  Working implementation
              │   (AI-generated, you        │  in your project, matching
              │    review)                  │  your existing patterns
              └─────────────────────────────┘
```

### Adobe's knowledge stream — what to build

Adobe provides three artifacts that describe the feature:

- **API Domain Docs** — the full Adobe VIPMP Partners API capability reference. Adobe engineers use this to author the API spec.
- **Experience Card** — the user journey: what the user does at each step, what API calls happen, what success and error states look like. Written by Adobe's PdMs.
- **API Spec** — every endpoint the feature calls: method, path, auth, request shape, response shape, error codes. Written by Adobe's engineers.

You receive the Experience Card and API Spec. You validate them for completeness before running any skills. You do not author any of these.

### Your knowledge stream — how to build it

- **Service Cards** — a structured description of how your specific codebase is built: your naming conventions, your auth patterns, your component structure, your error handling. Generated by the toolkit scanning your repo. You review and maintain them.

This is your primary contribution. The AI cannot write code that fits your project without accurate service cards.

### Where the streams merge — the AI's job

Once both streams are in place, the AI does the combining:

- **LLDs (Low-Level Designs)** — the AI reads Adobe's feature inputs and your service cards together to produce a feature plan tailored to your codebase. Two LLDs per feature: one backend, one UI. You review and approve both before any code is written.

- **Generated Code** — the AI executes the approved LLD, following the patterns in your service cards. You review the output as you would any engineer-authored PR.

### What this means in practice

Your most important job in this workflow is keeping your service cards accurate. If they correctly describe how your codebase works, the AI can produce code that looks like your team wrote it. If they are stale or inaccurate, the generated code will have pattern mismatches throughout — and you will spend more time fixing generated code than you would have spent writing it from scratch.

## Who Does What

There are two organizations involved in this workflow: **Adobe** and **you (the partner)**. Understanding this split is critical — you do not author the feature inputs, you validate them and then run the toolkit against your own codebase.

### Adobe provides (you receive and use)

**The toolkit itself (one time):**

Adobe ships this AI Kit — the skills that power every step of the workflow. You do not build or configure these; you install the kit and run the skills against your own codebases.

| Skill | What it does |
|---|---|
| `/generate-backend-service-card` | Scans your backend repo and produces service card files |
| `/generate-ui-service-card` | Scans your frontend repo and produces service card files |
| `/apply-api-spec` | Reads the API spec + your service cards → generates the backend LLD |
| `/apply-experience-card` | Reads the experience card + backend LLD + your service cards → generates the UI LLD |
| `/implement-feature` | Reads both approved LLDs + your service cards → generates working code |

**Per feature:**

| Artifact | Your role |
|---|---|
| Experience Card | Validate completeness; raise gaps before running any skills |
| API Spec | Validate completeness and consistency with experience card |

### You produce (with AI assistance)

| Artifact | How it's produced |
|---|---|
| Backend Service Cards | AI scans your backend repo | 
| UI Service Cards | AI scans your frontend repo | 
| Backend LLD | AI generates from API spec + service cards | 
| UI LLD | AI generates from experience card + backend LLD + service cards | 
| Generated Code | AI generates from LLDs + service cards |

## What the Toolkit Does Not Do

Be clear-eyed about the boundaries before starting.

- **Does not handle deployment or CI/CD.** Generated code lands in your repo. Deployment is your responsibility.
- **Does not guarantee production-ready code without review.** Engineer code review is mandatory, not optional.
- **Does not update service cards automatically.** When your codebase changes — new routes, new dependencies, new auth patterns — you must update the service cards to match. Generated code will follow outdated patterns if cards are stale.
- **Does not make product decisions.** The AI does not decide what the feature should do. That is the PM's job, expressed through the experience card.

## The Anytime Upgrade Example

Throughout this manual, every guide uses the **Anytime Upgrade** feature as a worked example. This feature allows a partner to upgrade a customer's subscription mid-term with prorated pricing.

The five-step user journey:
1. Check which upgrade paths are available for the customer's current subscription
2. Display the available upgrade options in the UI
3. Preview the upgrade order, including prorated pricing
4. Allow the user to adjust quantity and see the updated price
5. Place the switch order and confirm

This example is concrete enough to illustrate every phase of the workflow without being so complex that it obscures the process.
